Task 2 - Rules

  1. Create a policy/rule to block all incoming ping packets.
  2. Verify that the newly created rule/policy can correctly detect and block incoming ping packets from the attacker machine.

To create a new rule I have to add it in sudo nano /etc/snort/rules/rules/local.rules as shown below:

drop icmp any any -> $HOME_NET any (msg:"ICMP blocked"; sid:10000001; rev:001;)

It will drop any ICMP packets from 0.0.0.0/0 to any network port of the 10.2.1.0/24 subnet.

drop - Drop the packet using a software firewall and transfer the information to the logging system

To make it work I need to add a new iptables rule - sudo iptables --append FORWARD --jump NFQUEUE --queue-num 0.

The advantage of using numbered queues is that several queues can be created and the traffic of each queue is passed through a snort launched with a set of rules tailored to filtered traffic. A serious disadvantage of this approach is that in the event of a SNORT crash, the protected service becomes unavailable since there is no one to transfer data from the user-mode back to the kernel.

Let’s start the snort sudo snort -Q -c /etc/snort/etc/snort.conf -A console.

DoS

A DoS attack can be initiated by sending invalid data to applications or network services until the server hangs or simply crashes. The most common form of a DoS attack is TCP attacks.

Let’s run a tool called hping3 to launch a flooding attack - hping3 -c 10000 -d 120 -S -w 64 -p 21 --flood --rand-source 10.2.1.2.

It will send 10000 packets with a data size (-d) of 120 bytes to the address 10.2.1.2:21. It will set the TCP SYN flag for packets (-S) with TCP window size = 64 (-w).

– flood - Sent packets as fast as possible, without taking care to show incoming replies
–rand-source - will send packets with a random source address

Figure 9 shows the attack.


Figure 9 - DoS attack traffic

Now let’s create a new rule to prevent this.
drop tcp any any -> $HOME_NET any (msg:"DoS attack"; sid:10000002; rev: 002; detection_filter: track by_dst, count 50, seconds 20;)

detection_filter: track by_dst - snort tracks the destination IP address for detection.

seconds 20 - sampling period.

count 50 - if during the sampling period snort detects more than 50 requests then it will be dropped.

Bruteforce SSH

Brute force is a common attack. In my case, I tried this for ssh connection.

I created a rule to prevent it.

drop tcp any any -> $HOME_NET 22 (msg:"Potential SSH Brute Force Attack"; flow:to_server; flags:S; detection_filter: track by_src, count 5, seconds 60;sid:1000004; rev:2;).

It will be launched if snort detects more than 5 TCP packets on the 22nd port with the SYN flag within 60 seconds.

Task 3 - Questions

  1. Answer the following questions:
    1. What is a zero-day attack?
    2. Can IDS/IPS detect zero-day attack? if yes why? if no why?
    3. Given a network that has 1 million connections daily where 0.1% are attacks. If the IDS has a true positive rate of 95% what false positive alarm rate do we need to achieve to ensure the probability of an attack, given an alarm is 95%?

A zero-day attack exploits a zero-day vulnerability. This is a vulnerability that the developer does not yet know about, antivirus manufacturers or users do not know about. But the attacker knows. So IDS/IPS cannot directly detect such an attack. It can react to abnormal behavior or actions committed by an attacker.

0.1% - attacks
999 000 - TN and FP (false alarm or correct rejection)
1000 - true positive or false negative

1000 * 0,95 = 950 - true alarm

950/(950+x) = 0.05
x = 18050 (False alarm)

999 0000 * y = 18050

y = 0.01807 = 1.807%

False-positive alarm rate should be equal to 1.807%
